# kpi-tool プロジェクト ガイドライン

## コード変更後の自動レビュー手順

Edit ツールでコードを変更したら、必ず以下を確認してからユーザーに完了を報告すること。

### 1. 変数スコープの確認
- `const` / `let` を `if` / `for` などのブロック内で宣言した場合、そのブロック外で参照していないか確認する
- 特に `if (x) { const foo = ...; }` の後で `foo` を使っている箇所がないかチェックする
- 関数全体で使う変数は、関数の先頭で宣言する

### 2. 変更の影響範囲の確認
- 同じロジック（例：`isPlatinaPlus` 判定）が複数箇所にある場合、すべての箇所を同時に修正する
- Grep で変更したキーワードを検索し、見落としがないか確認する

### 3. 非同期処理・エラーハンドリングの確認
- `fetchSheet` など外部API呼び出しがエラー時に適切に処理されているか確認する
- `try-catch` の範囲内で不要な例外が発生していないか確認する

### 4. 月次タブについて
- `MONTHLY_SHEETS` は廃止済み。`monthlySheetName(offsetMonths)` で YYMM 形式を自動生成する
- 毎月新しいタブが追加されても、コード変更は不要（自動で最新月を参照する）
- 月次タブが存在しない場合は `fetchSheet` が空配列を返すので、ログインは壊れない

### 5. 代行設置フォームの同期ルール
- `delegate.html`（スタンドアロンページ）と `index.html` 内のモーダル（`#delegate-overlay`）は**同じフォーム内容**を保つ
- どちらか一方を修正したら、必ずもう一方にも同じ変更を反映すること
- 項目・文言・バリデーション・GASへのpayloadキー名を常に一致させること
- モーダル側はサロン情報が自動入力される点のみ異なる（delegate.htmlでは手動入力）

## プロジェクト構成
- 編集ファイル：`kpi-tool.html`（ローカル確認用）/ `index.html`（GitHub Pages 公開用）
- 変更は `index.html` に対して行い、auto-commit フックが自動コミットする
- 公開反映：`git push origin main` → GitHub Pages に数分で反映
